In praise of memcached

▲ 209 分 · j03b · 78 评论

文章以运维视角对比了 Redis 和 memcached，指出 Redis 常因功能丰富而被开发者误用作持久数据库，导致缓存层依赖过重与运维复杂；而 memcached 设计简洁、无持久化、客户端自动容错，更适合纯缓存场景，推荐优先使用它来降低维护负担。

  • Redis 虽能配置为缓存，但因开发者将其当作持久存储而引发生产事故，运维需额外监控和维持。
  • memcached 不提供持久化，客户端会自动忽略连接失败并返回默认值，集群通过客户端一致性哈希实现，节点宕机时自动剔除和恢复。
  • memcached 架构简单、资源开销低，可轻松将实例作为无状态工作负载调度，运维成本显著低于 Redis。
  • 大量“数据库太慢”问题实际源于查询未优化或缺少索引，引入缓存前应先检查查询性能。
  • memcached 所有操作均为 O(1) 设计，避免了因复杂操作造成的阻塞风险，而 Redis 单线程模型在执行复杂命令时可能导致延迟。

💬 讨论 评论区普遍认同 Redis 易被滥用为持久存储，赞同 memcached 在纯缓存场景下的简单可靠；但也有人担忧大团队最终仍会需要 Redis 的数据结构功能，导致需维护两种缓存技术，反增复杂性。另有评论提醒 Redis 已被 Valkey 取代，并强调 memcached 的 O(1) 操作保证了对稳定延迟的友好性。

The new HTTP QUERY method explained

▲ 183 分 · CommonGuy · 125 评论

RFC 10008 定义了新的 HTTP QUERY 方法，与 GET 类似但支持请求体，安全且幂等，用于解决 GET 无法携带复杂请求体、POST 语义不匹配等长期问题。

  • QUERY 方法被明确定义为安全、幂等，并支持缓存（需把请求体纳入缓存键）。
  • 使用 GET 携带请求体会因代理、防火墙等实现差异而导致不可靠，因此长期被回避。
  • 用 POST 做复杂查询虽能带请求体，但语义上非幂等，阻碍自动重试和中间件缓存。
  • 当前 QUERY 方法支持非常有限，全面部署可能需要数年，GET 查询在简单场景下仍完全够用。
  • QUERY 不适合需要分享或收藏链接的场景，因为其语义不依赖 URL 携带查询参数。

💬 讨论 评论区有人认为 QUERY 是对多年缺失的理性弥补，反对用 GET 加 body 的 hack；但也有人质疑其必要性，提出可用 POST 创建资源再 GET 的替代模式，或认为 HTTP 本身已显陈旧。部分人担忧中间设备对新方法的兼容性可能导致拒绝服务等问题。
